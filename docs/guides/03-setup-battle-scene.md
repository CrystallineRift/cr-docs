# Setting Up a Battle Scene

## Goal

By the end of this guide you will understand what the battle system needs in the scene, how `BattleCoordinator` is wired, and how to trigger both NPC and wild battles for testing.

## Prerequisites

- The `Scene Context` with `LocalDevGameInstaller` is in your scene.
- A trainer has been selected (i.e. `GameSessionManager.CurrentTrainerId` is non-null). Battles cannot start without an active trainer in session.
- For NPC battles: an NPC with `NpcTrainerBehaviour` is set up. See [Setting Up an NPC in a Scene](?page=guides/02-setup-npc-scene).
- For wild battles: a spawner with `SpawnerWorldBehaviour` and `SpawnerEncounterBehaviour` is set up (see below).

---

## How the Battle System is Wired

`BattleCoordinator` is **not placed manually in the scene**. It is created at runtime by Zenject from this binding in `LocalDevGameInstaller`:

```csharp
Container.Bind<IBattleCoordinator>()
    .To<BattleCoordinator>()
    .FromNewComponentOnNewGameObject()
    .AsSingle();
```

Zenject creates a new GameObject named `BattleCoordinator` and adds the `BattleCoordinator` MonoBehaviour to it. You will see it appear in the Hierarchy at runtime.

`BattleCoordinator` depends on:

| Injected type | What it does |
|---|---|
| `IBattleSystem` (→ `StatefulBattleSystemV2`) | Runs the actual turn-by-turn battle logic |
| `ICombatAI` (→ `BasicCombatAI`) | Controls the opponent's move selection |
| `GameSessionManager` | Reads `CurrentTrainerId` to identify the player |
| `ICRLogger` | Logging |

All four are bound in `LocalDevGameInstaller`. No manual wiring is needed beyond ensuring the installer is present.

## NPC Battle Flow

An NPC battle is triggered by `NpcInteractionBehaviour` when the player presses **E** near an NPC that has `NpcTrainerBehaviour` with `CanBattle == true`:

```
Player presses E
  → NpcInteractionBehaviour.Update()
    → BattleCoordinator.StartNpcBattle(NpcBattleRequest)
      → IBattleSystem.InitializeBattle(ONEvONE, playerShim, npcShim)
      → ICombatAI.InitializeAsync(npcTrainerId, ...)
      → BattleCoordinator.OnBattleStarted fires (when first player turn arrives)
```

`NpcBattleRequest` carries the NPC's trainer ID, creature team, and battle items — all resolved during `NpcWorldBehaviour` initialization.

No additional scene setup is needed for NPC battles beyond the NPC itself.

## Wild Battle Flow (Spawner Encounter)

Wild battles are triggered by `SpawnerEncounterBehaviour` when the player enters its trigger zone.

### Step 1 — Create the Spawner GameObject

1. Right-click in the Hierarchy under a `Spawners` group and choose **Create Empty**.
2. Name it descriptively, e.g. `GroveSpawner`.

### Step 2 — Attach SpawnerWorldBehaviour

1. Add `SpawnerWorldBehaviour` to the GameObject.
2. The `_spawnerContentKey` field is **auto-generated** when you add the component in the Editor — a GUID is written immediately via `OnValidate()`. You do not type a value here. Simply confirm it is non-empty after adding the component.

### Step 3 — Attach SpawnerEncounterBehaviour

`SpawnerEncounterBehaviour` requires `SpawnerWorldBehaviour` on the same GameObject.

1. Add `SpawnerEncounterBehaviour`.
2. A `SphereCollider` is added automatically at runtime by the script when the spawner activates. You do not need to add it manually.
3. Set the Inspector field:

| Field | Value |
|---|---|
| **Encounter Radius** | Default `5`. This is the radius (in Unity units) within which the player triggers a wild battle encounter. |

The spawner GameObject with both components looks like this in the Inspector:

```
GroveSpawner (GameObject)
├── Transform
├── SpawnerWorldBehaviour
│     Spawner Content Key: "<auto-generated GUID>"
└── SpawnerEncounterBehaviour
      Encounter Radius: 5
```

When `SpawnerWorldBehaviour.InitializeAsync()` completes, it calls `SpawnerEncounterBehaviour.Activate()`, which adds and enables the `SphereCollider`. Until that happens, the trigger is inactive.

## Battle State and Events

`IBattleCoordinator` exposes two events that UI or other systems can subscribe to:

| Event | Fires when |
|---|---|
| `OnBattleStarted` | The first player-turn event arrives from `IBattleSystem`. Carries a `BattleSession` with `PlayerTurnKey`. |
| `OnBattleEnded` | `BattleCoordinator.EndBattle()` is called. Carries a `BattleResult` with winner and reason. |

To receive these events, inject `IBattleCoordinator` into your own MonoBehaviour (via Zenject) and subscribe in the `[Inject]` method.

To end a battle manually (e.g. from a UI confirm button), call:

```csharp
_battleCoordinator.EndBattle(winnerTrainerId: null, reason: "forfeit");
```

## Triggering a Test Battle Without an NPC

For testing purposes you can call `BattleCoordinator` directly from a test MonoBehaviour. Inject `IBattleCoordinator` and call `StartNpcBattle` with a fabricated `NpcBattleRequest`. You still need a valid trainer ID from `GameSessionManager.CurrentTrainerId` — this requires a trainer to have been selected via the startup flow first.

## What Is Not Yet Implemented

The following battle scene elements are stub or incomplete as of the current source:

- **Battle UI**: `BattleCoordinator.HandlePlayerTurn` fires `OnBattleStarted` with the `BattleSession` but there is no UI scene wired to it yet. The `BattleSession.PlayerTurnKey` is intended to be passed to a turn-selection UI.
- **Camera transition**: No battle camera rig is currently in the scene. A battle camera setup is not implemented.
- **Win/loss outcome UI**: `OnBattleEnded` fires but no result screen is wired to it.

If you need to add UI for these, subscribe to `IBattleCoordinator.OnBattleStarted` and `OnBattleEnded` from a UI MonoBehaviour bound in the installer.

## Verification — Play Mode

1. Select a trainer via the startup flow so `GameSessionManager.CurrentTrainerId` is populated.
2. Walk the player character into an NPC's sphere or a spawner's encounter zone.
3. For NPC battle (press E near trainer NPC):
   ```
   [BattleCoordinator] NPC battle started. player=<guid> npc=<guid>
   ```
4. For wild battle (walk into spawner zone):
   ```
   [SpawnerEncounterBehaviour] 'GroveSpawner' — player entered zone, starting wild battle
   [BattleCoordinator] Wild battle started. player=<guid> wildTrainer=<guid> spawner=<guid>
   ```

## Common Mistakes

**"StartNpcBattle called while battle already in progress — ignoring."**
A battle was started before the previous one ended. Call `BattleCoordinator.EndBattle()` before starting a new battle, or ensure `OnBattleEnded` fires and is handled by the caller.

**"no current trainer in session, cannot start battle."**
No trainer has been selected. The startup flow (character select screen) must call `GameSessionManager.SetTrainerAsync()` before any battle can begin.

**Spawner encounter zone never activates.**
`SpawnerWorldBehaviour.InitializeAsync()` did not complete (check logs for spawner init errors), or `WorldRegistry` was empty when `GameInitializer` ran. Confirm the spawner GameObject was present at scene load time, not instantiated later.

**`SpawnerEncounterBehaviour` triggers but `ICombatAI` errors appear.**
The wild trainer ID used as the opponent is the spawner's ID (a proxy). If `BasicCombatAI` cannot load team data for that ID, it will log an error. This is expected until the spawner-to-wild-trainer mapping is fully implemented in the backend.
