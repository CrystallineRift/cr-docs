# NPC System

NPCs are world entities scoped to a specific trainer's world. Each NPC can hold a team of up to six creatures and optionally give a starter creature to the player. The NPC system is one of the most actively used domains during early gameplay — every new trainer triggers the starter NPC flow on their first world load.

## Why This Design?

### Why Is Each NPC Scoped to a Trainer?

In Crystalline Rift, each trainer has their own persistent world. Two different trainers logging into the same account do not share NPCs — each trainer's world is initialized independently with its own NPC instances. This scoping (always requiring both `accountId` and `trainerId` on every query) prevents accidental cross-trainer data leakage and makes the data model explicit about ownership.

### Why `content_id` vs `id`?

Every NPC row carries two identifiers:
- `id` (UUID) — generated by the database at insert time, used for all internal FK relationships
- `content_id` (UUID) — set by seed data or the Unity Inspector, stable across sessions and environments

The `content_id` is the bridge between the backend database and Unity's asset system. When Unity's `NpcWorldBehaviour` wakes up, it reads `_npcContentId` from the Inspector — a GUID that was placed there by the level designer when they placed the NPC GameObject. The backend's `EnsureStarterNpcAsync` uses that `content_id` to look up whether this NPC already exists for this trainer, creating it if not.

This design means designers can work entirely in Unity's Inspector and YAML config files without needing to know or modify database UUIDs. The `content_key` string (e.g., `"cindris_starter_npc"`) is an additional human-readable label for debugging and seed scripts.

### Why Is `EnsureStarterNpc` Idempotent?

The starter NPC initialization runs every time the player loads the world. On the very first load, the NPC needs to be created and stocked with a starter creature. On every subsequent load, it just needs to return the existing NPC's state. Making this operation idempotent (safe to call multiple times) avoids the need for the Unity client to track "have I initialized this NPC?" across sessions, app restarts, or crashes.

The backend simply checks: "does an NPC with this `content_id` exist for this trainer?" If yes, return it. If no, create it. No locks, no session flags, no client-side state required.

## Data Model

| Column | Type | Notes |
|--------|------|-------|
| `id` | UUID | Generated primary key |
| `content_id` | UUID | Stable identifier set by seed data / Unity Inspector |
| `content_key` | STRING | Human-readable key (e.g. `"cindris_starter_npc"`) |
| `trainer_id` | UUID | Owning trainer's world |
| `account_id` | UUID | Owning account |
| `npc_type` | STRING | `"starter"`, `"merchant"`, `"guide"`, etc. |
| `name` | STRING | Display name |
| `creature_team_inventory_id` | UUID | FK → `inventory` table (the NPC's team) |
| `deleted` | BOOLEAN | Soft delete |
| `created_at` | TIMESTAMP | Audit trail |
| `updated_at` | TIMESTAMP | Audit trail |

Each NPC also has an associated **team inventory** (`npc_creature_team`) with up to 6 slots. The inventory is created atomically with the NPC row inside a database transaction — if the inventory creation fails, the NPC row is rolled back. This is enforced in `NpcDomainService.CreateNpcAsync` which opens a connection, begins a transaction, inserts the NPC, creates the inventory, updates the NPC with the inventory ID, then commits.

## Internals: CreateNpcAsync Transaction

The creation of an NPC is a three-step atomic transaction:

```
BEGIN TRANSACTION
  1. INSERT npc row (without creature_team_inventory_id)
  2. INSERT inventory row (type=CreatureTeam, maxSlots=6)
  3. UPDATE npc SET creature_team_inventory_id = <new_inventory_id>
COMMIT
```

If any step fails, the transaction is rolled back and neither the NPC row nor the inventory row persists. This prevents orphaned inventory rows (inventory exists, NPC does not) or NPC rows with null inventory IDs. The method signatures for the transactional variants — `CreateNpcInTransactionAsync`, `CreateInventoryInTransactionAsync`, `UpdateNpcInTransactionAsync` — accept the `IDbTransaction` explicitly so the repository layer participates in the caller's transaction without owning connection lifecycle.

## Key Domain Operations

Source: `Npcs/CR.Npcs.Domain.Services/Interface/INpcDomainService.cs`

### `EnsureStarterNpcAsync`

Idempotent upsert — creates the NPC and loads its team with a starter creature on first call, returns the existing NPC on subsequent calls.

```csharp
Task<NpcBase> EnsureStarterNpcAsync(
    Guid accountId,
    Guid trainerId,
    Guid contentId,              // stable ID from Unity Inspector
    Guid starterCreatureBaseId,  // base creature to generate on first run
    CancellationToken ct = default);
```

Internally, `EnsureStarterNpcAsync` in `NpcDomainService`:
1. Calls `GetNpcByContentIdAsync` — if found, returns immediately
2. Builds a `NpcBase` with `NpcType = NpcType.Guide`
3. Calls `CreateNpcAsync` (which opens a transaction for NPC + inventory creation)
4. Calls `ICreatureGenerationService.GetAvailableGrowthProfilesAsync` — throws `InvalidOperationException` if no growth profiles exist for the base creature
5. Calls `ICreatureGenerationService.GetAvailableAbilityProgressionSetsAsync` — uses first set if any, null if none
6. Calls `ICreatureGenerationService.CreateAsync` with `Level = 1`, `Nature.Hardy`, `Gender.Unknown`
7. Calls `AddCreatureToNpcTeamAsync` at slot 1

**Failure mode:** If `starterCreatureBaseId` references a creature with no growth profiles in the database, step 4 throws and the NPC row is not created (the transaction around step 3 already committed, so the NPC row will exist without a creature). This is a data integrity issue — always ensure the base creature has at least one growth profile before using it in `EnsureStarterNpc`. See [Creature Generation](?page=backend/04-creature-generation) for how to set up growth profiles.

### `GiveNpcCreatureToTrainerStorageAsync`

Transfers the first creature from the NPC's team into the trainer's storage inventory. Fails if the NPC has no creatures.

```csharp
Task<GeneratedCreature> GiveNpcCreatureToTrainerStorageAsync(
    Guid npcId,
    Guid accountId,
    Guid trainerId,
    CancellationToken ct = default);
```

Internally:
1. Fetches the NPC's team — throws `InvalidOperationException` if empty
2. Takes `team.First()` (slot 1 in practice)
3. Removes the creature from the NPC's team
4. Gets or creates the trainer's `Creature` storage inventory (creates a 100-slot inventory named "Storage" if none exists)
5. Determines the next available slot (max existing slot + 1, or 1 if empty)
6. Adds the creature to the trainer's storage inventory
7. Updates the creature's `CurrentTrainerId` to the receiving trainer

**Failure mode:** If the trainer's storage inventory is full (more than 100 creatures), slot assignment continues incrementally beyond `MaxSlots`. There is currently no hard enforcement of `MaxSlots` at the repository level — it is advisory metadata. A future enforcement pass should add a capacity check before step 5.

### `GetNpcByContentIdAsync`

```csharp
Task<NpcBase?> GetNpcByContentIdAsync(
    Guid accountId, Guid trainerId, Guid contentId, CancellationToken ct = default);
```

Always scoped by `accountId` + `trainerId`. Returns `null` if not found (this is the signal that `EnsureStarterNpc` uses to decide whether to create). All three IDs must be non-empty GUIDs or an `ArgumentException` is thrown before any DB call is made — this is validated in both the interface contract and the implementation.

## REST Endpoints

All NPC endpoints are prefixed `/api/v1/npc`.

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/api/v1/npc/ensure-starter` | `EnsureStarterNpc` — idempotent upsert |
| `POST` | `/api/v1/npc/{id}/give-creature` | Give NPC's creature to trainer storage |
| `GET` | `/api/v1/npc/{id}` | Get NPC by ID |
| `GET` | `/api/v1/npc` | List NPCs for trainer |
| `POST` | `/api/v1/npc` | Create NPC |
| `PUT` | `/api/v1/npc/{id}` | Update NPC |
| `DELETE` | `/api/v1/npc/{id}` | Soft-delete NPC |

### EnsureStarterNpc request/response

```json
POST /api/v1/npc/ensure-starter
{
  "accountId":             "00000000-...",
  "trainerId":             "00000000-...",
  "contentId":             "aabb1234-...",   // from Unity Inspector
  "starterCreatureBaseId": "ccdd5678-..."    // base creature UUID
}

→ 200 OK
{
  "npcId":           "ffee9012-...",
  "hasCreatureToGive": true
}
```

`hasCreatureToGive` is `true` when `GetNpcTeamAsync` returns at least one creature. After the player calls `give-creature`, the next `ensure-starter` call returns `hasCreatureToGive: false`.

## Common Mistakes / Tips

- **Missing content_id in Unity Inspector.** If `_npcContentId` is empty or not a valid GUID, `NpcWorldBehaviour.InitializeAsync` returns early without calling the backend. The NPC will appear in the scene but never initialize. Always verify the Inspector field is set and parses as a GUID.
- **Duplicate content_id across trainers.** Two different trainers can each have their own NPC with the same `content_id` — this is correct and expected (same NPC definition, different trainer worlds). Do not confuse this with a collision.
- **Calling `give-creature` before `ensure-starter` completes.** `NpcInteractionBehaviour` checks `HasCreatureToGive` before allowing the interaction. If world init fails silently, `HasCreatureToGive` stays false. Check backend logs for the `ensure-starter` POST to confirm it succeeded.
- **base creature has no growth profiles.** `EnsureStarterNpcAsync` throws on step 4 if `GetAvailableGrowthProfilesAsync` returns empty. Seed the `growth_profile` table before seeding NPCs.
- **Slot conflicts on team add.** `AddCreatureToNpcTeamAsync` throws `InvalidOperationException` if the target slot is already occupied. When seeding NPCs with pre-built teams, add creatures in slot order starting from 1.
