# NPC System

NPCs are world entities scoped to a specific trainer's world. Each NPC can hold a team of up to six creatures and an item inventory. The NPC system is one of the most actively used domains during early gameplay — every new trainer triggers the starter NPC flow on their first world load.

## Why This Design?

### Why Is Each NPC Scoped to a Trainer?

In Crystalline Rift, each trainer has their own persistent world. Two different trainers logging into the same account do not share NPCs — each trainer's world is initialized independently with its own NPC instances. This scoping (always requiring both `accountId` and `trainerId` on every query) prevents accidental cross-trainer data leakage and makes the data model explicit about ownership.

### Why `content_key` vs `id`?

Every NPC row carries two identifiers:
- `id` (UUID) — generated by the database at insert time, used for all internal FK relationships
- `content_key` (string) — set by seed data or the Unity Inspector, stable across sessions and environments (e.g. `"cindris_starter_npc"`)

The `content_key` is the bridge between the backend database and Unity's asset system. When Unity's `NpcWorldBehaviour` wakes up, it reads `_npcContentKey` from the Inspector — a string that was placed there by the level designer when they placed the NPC GameObject. The backend's `EnsureNpcAsync` uses that `content_key` to look up whether this NPC already exists for this trainer, creating it if not.

This design means designers can work entirely in Unity's Inspector and YAML config files without needing to know or modify database UUIDs. String keys are also readable in migrations, seed scripts, logs, and the Unity inspector — no GUID parsing required.

The `content_key` is **unique per trainer** — the database enforces a unique index on `(account_id, trainer_id, content_key)` (where `content_key IS NOT NULL`).

### Why Are Ensure Operations Idempotent?

The NPC initialization flow runs every time the player loads the world. On the very first load, the NPC needs to be created and optionally stocked with a creature team and items. On every subsequent load, it just needs to return the existing NPC's state. Making these operations idempotent (safe to call multiple times) avoids the need for the Unity client to track "have I initialized this NPC?" across sessions, app restarts, or crashes.

`EnsureNpcAsync` checks: "does an NPC with this `content_key` exist for this trainer?" If yes, return it. If no, create it. `EnsureNpcCreatureTeamAsync` extends this per slot — each slot is only filled if it is currently empty. `EnsureNpcItemsAsync` extends this per itemId — each item is only added if it is not already present.

### Why Random Seeds Per Creature Instead of a Fixed Seed?

`EnsureNpcCreatureTeamAsync` uses `Guid.NewGuid().GetHashCode()` as the seed for each creature slot rather than a fixed seed (e.g. `0`). This ensures each trainer's instance of an NPC has a uniquely-rolled team — two trainers interacting with the same NPC type in the same region get different individual creature variants. This is intentional game design: NPC teams are not identical across all players.

### Why Does `NpcDomainService` Require `IDbConnectionFactory`?

`CreateNpcAsync` opens a database connection and manually manages a transaction to atomically create the NPC row and its team inventory together. This is why `NpcDomainService` must be registered as `AddScoped` (not `AddSingleton`) — `IDbConnectionFactory` is scoped per HTTP request. Injecting a scoped service into a singleton would capture the wrong lifetime.

## Data Model

| Column | Type | Notes |
|--------|------|-------|
| `id` | UUID | Generated primary key |
| `content_key` | STRING | Stable designer-facing key (e.g. `"cindris_starter_npc"`); unique per trainer when set |
| `trainer_id` | UUID | Owning trainer's world |
| `account_id` | UUID | Owning account |
| `npc_type` | INT16 | Enum: `Npc=2`, `Merchant`, `Trainer`, `Guide`, etc. |
| `name` | STRING | Display name (defaults to `content_key` on creation via `EnsureNpcAsync`) |
| `creature_team_inventory_id` | UUID | FK → `inventory` table (the NPC's team) |
| `deleted` | BOOLEAN | Soft delete |
| `created_at` | TIMESTAMP | Audit trail |
| `updated_at` | TIMESTAMP | Audit trail |

Each NPC also has an associated **team inventory** (`npc_creature_team`) with up to 6 slots and a separate **item inventory** (`npc_inventory`). The team inventory is created atomically with the NPC row inside a database transaction — if the inventory creation fails, the NPC row is rolled back.

### `NpcCreatureSlotSpec` Value Type

`NpcCreatureSlotSpec` is a value type that pairs a creature base reference with its target team slot:

```csharp
public record NpcCreatureSlotSpec(string CreatureBaseContentKey, int SlotNumber);
```

It is the input element type for `EnsureNpcCreatureTeamAsync`. Each spec drives one slot — the `CreatureBaseContentKey` identifies what kind of creature to generate (looked up via `ICreatureRepository.GetByContentKeyAsync`) and `SlotNumber` (1–6) identifies where it goes in the NPC's team. Passing duplicate slot numbers within a single call is a caller error; the service processes slots in order and the second attempt on an already-filled slot is silently skipped (idempotent per slot).

### `NpcItemSeedSpec` Value Type

```csharp
public class NpcItemSeedSpec
{
    public NpcItemSeedSpec(Guid itemId, int quantity) { ItemId = itemId; Quantity = quantity; }
    public Guid ItemId { get; }
    public int Quantity { get; }
}
```

It is the input element type for `EnsureNpcItemsAsync`. Each spec drives one item entry — idempotent per `ItemId`. If an item with the given `ItemId` already exists in the NPC's inventory, the spec is skipped entirely (quantity is NOT incremented). The initial quantity is only written on first seed.

## Internals: CreateNpcAsync Transaction

The creation of an NPC is a three-step atomic transaction. From the actual implementation in `NpcDomainService.CreateNpcAsync`:

```csharp
await using var connection = CreateConnection();
await connection.OpenAsync(ct);
await using var transaction = await connection.BeginTransactionAsync(ct);

try
{
    var npcId = await _npcRepository.CreateNpcInTransactionAsync(
        accountId, trainerId, npc, transaction);
    npc.Id = npcId;

    var creatureTeamInventory = await _trainerInventoryRepository.CreateInventoryInTransactionAsync(
        accountId, trainerId, new Inventory
        {
            Name = $"NPC {npc.Name} Team",
            Type = InventoryType.CreatureTeam,
            MaxSlots = 6,
            AccountId = accountId,
            TrainerId = trainerId
        }, transaction);
    npc.CreatureTeamInventoryId = creatureTeamInventory.Id;

    await _npcRepository.UpdateNpcInTransactionAsync(
        accountId, trainerId, npcId, npc, transaction);

    await transaction.CommitAsync(ct);
    return npc;
}
catch
{
    await transaction.RollbackAsync(ct);
    throw;
}
```

If any step fails, the transaction is rolled back and neither the NPC row nor the inventory row persists. The `*InTransactionAsync` variants accept the `IDbTransaction` explicitly so the repository layer participates in the caller's transaction without owning connection lifecycle.

## Full Usage Walkthrough

This is the typical sequence Unity follows during world initialization for an NPC that hands out a starter creature:

### Step 1 — Ensure the NPC exists

```
POST /api/v1/npc/ensure-starter
{
  "accountId":             "aaaaaaaa-0000-0000-0000-000000000001",
  "trainerId":             "bbbbbbbb-0000-0000-0000-000000000002",
  "contentKey":            "cindris_starter_npc",
  "starterCreatureContentKey": "cindris"
}

→ 200 OK
{
  "npcId":             "dddddddd-0000-0000-0000-000000000004",
  "hasCreatureToGive": true
}
```

On the first call, this creates the NPC row + team inventory + one generated creature at slot 1. On all subsequent calls it returns the existing NPC immediately. `hasCreatureToGive` is `true` as long as the NPC's team is non-empty.

### Step 2 — Fetch the NPC

```
GET /api/v1/npc/dddddddd-0000-0000-0000-000000000004
  ?accountId=aaaaaaaa-0000-0000-0000-000000000001
  &trainerId=bbbbbbbb-0000-0000-0000-000000000002

→ 200 OK
{
  "id":        "dddddddd-...",
  "contentKey": "cindris_starter_npc",
  "npcType":   2,
  "name":      "cindris_starter_npc",
  ...
}
```

### Step 3 — Seed the NPC's item inventory (optional)

```
POST /api/v1/npc/dddddddd-.../ensure-items
{
  "accountId": "aaaaaaaa-...",
  "trainerId": "bbbbbbbb-...",
  "items": [
    { "itemId": "eeeeeeee-0000-0000-0000-000000000005", "quantity": 3 },
    { "itemId": "ffffffff-0000-0000-0000-000000000006", "quantity": 1 }
  ]
}

→ 200 OK
{ "itemCount": 2 }
```

Safe to call on every world load — items already in the inventory are silently skipped.

### Step 4 — Fetch the NPC's items

```
GET /api/v1/npc/dddddddd-.../items
  ?accountId=aaaaaaaa-...&trainerId=bbbbbbbb-...

→ 200 OK
{
  "items": [
    { "npcId": "dddddddd-...", "itemId": "eeeeeeee-...", "quantity": 3 },
    { "npcId": "dddddddd-...", "itemId": "ffffffff-...", "quantity": 1 }
  ]
}
```

### Step 5 — Player receives the creature

```
POST /api/v1/npc/dddddddd-.../give-creature
{
  "accountId": "aaaaaaaa-...",
  "trainerId": "bbbbbbbb-..."
}

→ 200 OK
{
  "creatureId":   "12121212-...",
  "creatureName": "cindris_starter_npc"
}
```

After this call, the NPC's team is empty. The next `ensure-starter` call returns `hasCreatureToGive: false`.

## How to Test NPC Endpoints Locally (curl)

Start the server with `cd Convenience/CR.REST.AIO && dotnet run`, then:

```bash
# 1. Register an account and get a token
TOKEN=$(curl -s -X POST http://localhost:5000/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","password":"test1234"}' \
  | jq -r '.accessToken')

# 2. Create a trainer (get trainerId from response)
TRAINER_ID=$(curl -s -X POST http://localhost:5000/api/v1/trainers \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"TestTrainer","accountId":"<accountId>"}' \
  | jq -r '.id')

# 3. Ensure the starter NPC
curl -s -X POST http://localhost:5000/api/v1/npc/ensure-starter \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{
    \"accountId\": \"<accountId>\",
    \"trainerId\": \"$TRAINER_ID\",
    \"contentKey\": \"cindris_starter_npc\",
    \"starterCreatureContentKey\": \"cindris\"
  }"
```

The `accountId` comes from the register response. Use `jq` to extract it alongside the token. The `starterCreatureContentKey` must match the `content_key` column in the `creature` table — seeded values include `"cindris"`, `"starter_1"`, `"starter_2"`, `"starter_3"`.

## Key Domain Operations

Source: `Npcs/CR.Npcs.Domain.Services/Interface/INpcDomainService.cs`

### `EnsureNpcAsync`

Idempotent bare NPC creation — creates the NPC row (with its team inventory) on first call, returns the existing NPC on subsequent calls. Does **not** populate the creature team.

```csharp
Task<NpcBase> EnsureNpcAsync(
    Guid accountId,
    Guid trainerId,
    string contentKey,
    NpcType npcType = NpcType.Npc,
    CancellationToken ct = default);
```

The `npcType` parameter controls what type is written when the NPC is **first created**. It is a **first-write-wins** field: if the NPC already exists, its type is returned as-is and `npcType` is ignored.

Internally:
1. Calls `GetNpcByContentKeyAsync` — if found, returns immediately (type not updated)
2. Builds a `NpcBase` with `Name = contentKey` (the name defaults to the content key on creation)
3. Calls `CreateNpcAsync` (transactional: NPC row + empty team inventory)

### `EnsureNpcCreatureTeamAsync`

Idempotent multi-slot creature team seeding. For each `NpcCreatureSlotSpec` in `slots`, fills that slot only if it is currently empty. Uses `Guid.NewGuid().GetHashCode()` as the random seed per creature.

```csharp
Task<int> EnsureNpcCreatureTeamAsync(
    Guid npcId,
    Guid accountId,
    Guid trainerId,
    IReadOnlyList<NpcCreatureSlotSpec> slots,
    CancellationToken ct = default);
```

Returns the final team count (total creatures currently in team, including pre-existing ones).

Internally, for each slot:
1. Checks whether the NPC's team already has a creature in that slot
2. If occupied — skips (idempotent)
3. If empty — resolves the `BaseCreature` via `ICreatureRepository.GetByContentKeyAsync(slot.CreatureBaseContentKey)`, then calls `ICreatureGenerationService.GetAvailableGrowthProfilesAsync` for the resolved creature
4. Calls `ICreatureGenerationService.GetAvailableAbilityProgressionSetsAsync` — uses the first set if any exist
5. Calls `ICreatureGenerationService.CreateAsync` with level=1, `Nature.Hardy`, `Gender.Unknown`, and a random seed
6. Calls `AddCreatureToNpcTeamAsync` at the target slot number

**Failure mode:** If `CreatureBaseContentKey` does not match any row in the `creature` table, `GetByContentKeyAsync` returns `null` and step 3 throws `InvalidOperationException`. If the resolved creature has no growth profiles, step 3 also throws `InvalidOperationException`. The slot is not filled. Always ensure base creatures have at least one growth profile. See [Creature Generation](?page=backend/04-creature-generation).

### `EnsureNpcItemsAsync`

Idempotent item inventory seeding. For each `NpcItemSeedSpec` in `items`, adds the item to the NPC's inventory only if an entry with that `ItemId` does not yet exist.

```csharp
Task<int> EnsureNpcItemsAsync(
    Guid npcId,
    Guid accountId,
    Guid trainerId,
    IReadOnlyList<NpcItemSeedSpec> items,
    CancellationToken ct = default);
```

Returns the final item count in the NPC's inventory after the operation.

- **Does NOT increment quantity** if an item already exists. First-seed wins.
- If all items already exist, no writes occur and the current count is returned.
- Designed to be called on every world load — safe to call repeatedly.

### `GetNpcItemsAsync`

Fetches the NPC's current item inventory.

```csharp
Task<IReadOnlyList<NpcInventoryEntry>> GetNpcItemsAsync(
    Guid npcId,
    Guid accountId,
    Guid trainerId,
    CancellationToken ct = default);
```

Delegates directly to `INpcInventoryRepository.GetItems`. Returns an empty list if the NPC has no items.

### `EnsureStarterNpcAsync`

Delegates to `EnsureNpcAsync` + `EnsureNpcCreatureTeamAsync`. Exists as a single convenience entry point for the starter NPC flow.

```csharp
Task<NpcBase> EnsureStarterNpcAsync(
    Guid accountId,
    Guid trainerId,
    string contentKey,
    string starterCreatureContentKey,
    CancellationToken ct = default);
```

Internally:
1. Calls `EnsureNpcAsync(accountId, trainerId, contentKey)` — ensures the NPC row exists (defaults to `NpcType.Npc`)
2. Calls `EnsureNpcCreatureTeamAsync(npc.Id, accountId, trainerId, [new NpcCreatureSlotSpec(starterCreatureContentKey, 1)])` — fills slot 1 if empty

### `GiveNpcCreatureToTrainerStorageAsync`

Transfers the first creature from the NPC's team into the trainer's storage inventory. Fails if the NPC has no creatures.

```csharp
Task<GeneratedCreature> GiveNpcCreatureToTrainerStorageAsync(
    Guid npcId,
    Guid accountId,
    Guid trainerId,
    CancellationToken ct = default);
```

Internally:
1. Fetches the NPC's team — throws `InvalidOperationException` if empty
2. Takes `team.First()` (slot 1 in practice)
3. Removes the creature from the NPC's team
4. Gets or creates the trainer's `Creature` storage inventory (creates a 100-slot inventory named `"Storage"` if none exists)
5. Determines the next available slot (`existingSlots.Max(c => c.SlotNumber) + 1`, or 1 if empty)
6. Adds the creature to the trainer's storage inventory
7. Updates the creature's `CurrentTrainerId` to the receiving trainer

**Failure mode:** If the trainer's storage inventory is full (more than 100 creatures), slot assignment continues incrementally beyond `MaxSlots`. There is currently no hard enforcement of `MaxSlots` at the repository level — it is advisory metadata.

## Common Mistakes / Tips

- **Missing content_key in Unity Inspector.** If `_npcContentKey` is empty or whitespace, `NpcWorldBehaviour.InitializeAsync` returns early without calling the backend. The NPC will appear in the scene but never initialize. Always verify the Inspector field is set.
- **Duplicate content_key across trainers is intentional.** Two different trainers can each have their own NPC with the same `content_key` — this is correct and expected (same NPC definition, different trainer worlds). The uniqueness constraint is per `(account_id, trainer_id, content_key)`, not globally unique.
- **Calling `give-creature` before `ensure-starter` completes.** `NpcInteractionBehaviour` checks `HasCreatureToGive` before allowing the interaction. If world init fails silently, `HasCreatureToGive` stays false. Check backend logs for the `ensure-starter` POST to confirm it succeeded.
- **Base creature has no growth profiles.** `EnsureNpcCreatureTeamAsync` throws on step 3 if `GetAvailableGrowthProfilesAsync` returns empty. Seed the `growth_profile` table before seeding NPCs.
- **Slot conflicts on team add.** `AddCreatureToNpcTeamAsync` throws `InvalidOperationException` if the target slot is already occupied and `slotNumber` is outside 1–6. `EnsureNpcCreatureTeamAsync` avoids this by checking occupancy first — only call `AddCreatureToNpcTeamAsync` directly if you own the idempotency check.
- **Expecting fixed-seed creatures across trainers.** `EnsureNpcCreatureTeamAsync` uses `Guid.NewGuid().GetHashCode()` as the seed by design. Do not rely on NPC teams being identical between different trainers — each trainer gets a uniquely rolled team.
- **`EnsureNpcItemsAsync` does not increment quantity.** If an item already exists in the NPC's inventory, calling `EnsureNpcItemsAsync` again with a higher quantity spec does nothing. Quantities should only be managed explicitly via `UpdateItemQuantity`. Seed specs are "set once" values.
- **`INpcInventoryRepository` must be registered.** `NpcDomainService` requires `INpcInventoryRepository` via constructor injection. Forgetting the `AddSingleton<INpcInventoryRepository>` registration in `Program.cs` causes a runtime DI failure when `INpcDomainService` is first resolved.
- **`NpcDomainService` registered as `AddSingleton`.** It injects `IDbConnectionFactory` which is `AddScoped`. Registering the domain service as `Singleton` will cause either a DI validation error at startup (if `ValidateScopes` is enabled) or a deadlock if it is not. Always use `AddScoped<INpcDomainService, NpcDomainService>()`.
